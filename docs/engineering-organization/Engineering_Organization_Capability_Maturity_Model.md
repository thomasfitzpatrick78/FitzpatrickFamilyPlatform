# Engineering Organization Capability Maturity Model

**Document Version:** 1.0

**Status:** Active

**Milestone:** EO-13.1

---

## Purpose

This document defines practical maturity criteria for the AI-operated Engineering Organization.

It provides assessment guidance only. It does not assign maturity scores unless evidence is captured during a governed review.

---

## Maturity Levels

| Level | Name | Criteria |
|-------|------|----------|
| Level 0 | Ad hoc | Work is mostly informal, inconsistent, or conversation-bound. Evidence is incomplete. |
| Level 1 | Defined | The capability has documented intent, ownership, and basic repository artifacts. |
| Level 2 | Repeatable | The capability has been used successfully across more than one workstream or milestone. |
| Level 3 | Governed | The capability has approved artifacts, review gates, authority boundaries, and validation expectations. |
| Level 4 | Measured | The capability has evidence, metrics, trend review, and recurring assessment. |
| Level 5 | Continuously improving | The capability uses evidence to improve practices, governance, automation, and roadmap direction. |

---

## Capability Areas

The following Engineering Organization capabilities should be assessed at milestone closeout or another approved governance cadence:

| Capability | Evidence Expectations |
|------------|-----------------------|
| Architecture governance | ADRs, architecture options, selection rationale, review outcomes, and explicit current-state versus planned-state separation. |
| Product governance | Vision, capability model, roadmap, backlog, milestone planning, and customer-value traceability. |
| Requirements quality | Approved requirements, acceptance criteria, non-goals, dependencies, and scope boundaries. |
| Repository governance | Indexed artifacts, valid links, clean repository hygiene, and durable decision records. |
| Delivery automation | Tested EAP capabilities, deterministic validators, generated reports, and documented limits. |
| Release management | Definition of Done evidence, readiness reports, milestone state, and authorized release actions. |
| AI role coordination | Role catalog entries, authority boundaries, escalation conditions, and handoff evidence. |
| Live execution safety | Human approval gates, stop conditions, rollback paths, and no independent scope expansion by execution agents. |
| Observability | Telemetry scope, dashboards, evidence templates, operational validation, and known gaps. |
| Incident response | Severity model, runbooks, escalation paths, evidence capture, and post-incident learning. |
| Security governance | Least privilege, privileged integration review, secrets hygiene, and trust-boundary documentation. |
| Evidence management | Reports, operational evidence, validation outputs, and repository-preserved review records. |
| Knowledge management | Engineering memory, documentation indexes, promoted practices, and transition packages. |
| Customer delivery | Customer-facing capability movement, application evidence, and value outcomes. |
| Engineering metrics | Repository, governance, release, closeout, and capability metrics with trend interpretation. |
| Capability planning | Coordinated EO, PLAT, and application roadmap streams with milestone-level investment traceability. |

---

## Assessment Guidance

Assessments must:

- cite repository evidence instead of relying on recollection;
- distinguish current state from planned state;
- identify the maturity level only where evidence supports it;
- record improvement opportunities and roadmap impacts;
- disclose exceptions to the Engineering Investment Rule;
- avoid claiming production readiness without operational evidence.
